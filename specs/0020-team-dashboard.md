# 0020 — Team Savings Dashboard

> 在现有单用户 CLI 成本与压缩统计能力之上，提供面向团队管理者的多维度节省分析与预算控制能力，作为 tokenless 商业化的核心团队功能入口。

---

## 1. 背景与目标用户

### 1.1 背景

当前 tokenless 已具备本地统计能力：CLI 与 hook 执行过程能够记录 `project`、`agent`、`session` 等维度的压缩与节省数据，并通过 `stats.db` 持久化。与此同时，`0018` 已定义 token 与 API 成本联动的方向，使“节省了多少 token”进一步能够映射为“节省了多少真实费用”。

但现阶段能力仍以**单用户、单机、命令行查看**为主，难以支撑团队级场景：

- 团队管理者无法跨项目查看整体节省贡献
- DevOps 无法识别异常峰值、预算超支风险或某类工具的异常成本
- CTO 无法量化 tokenless 在多个成员、多条业务线上的 ROI
- 商业化场景下，缺少一个可演示、可导出、可汇报的团队级价值面板

因此需要新增 **Team Savings Dashboard**，把已有统计能力提升为“团队成本控制台”：

- 面向管理视角展示节省与成本
- 面向运营视角提供趋势、异常与预算信号
- 面向商业化场景输出可演示、可导出的报表能力

### 1.2 目标用户

#### 团队管理者

关注团队整体是否真正节省了成本，谁在贡献节省，哪些项目最值得继续推广 tokenless。

核心问题：

- 团队本周/本月总共节省了多少 token 与费用
- 哪些成员、项目、agent 带来了最多节省
- 节省是否稳定，还是偶发性的

#### DevOps / Platform / FinOps

关注运行成本、峰值波动、预算控制与容量规划。

核心问题：

- 哪个项目或会话在短时间内出现异常 token 峰值
- 某类工具调用是否显著抬高 API 成本
- 预算是否接近阈值，需要提前预警

#### CTO / Engineering Leadership

关注商业价值、组织级 ROI 与部署决策。

核心问题：

- tokenless 在多团队、多项目中的净节省是否足够显著
- 是否应把 tokenless 从个体工具升级为团队标准平台
- 是否值得引入集中式服务或 SaaS 托管能力

---

## 2. 产品目标与非目标

### 2.1 目标

1. 在现有统计数据基础上，提供团队级聚合视图
2. 支持以成员、会话、工具、项目、agent 等维度查看节省
3. 将 token 节省映射为 API 费用节省，提供统一货币口径
4. 提供周/月趋势图，支持管理汇报与持续跟踪
5. 提供异常峰值检测与预算预警，支持运维治理
6. 以分阶段方式落地，从本地离线报告演进到本地服务，再演进到 SaaS

### 2.2 非目标

以下能力不属于本 spec 首期目标：

- 不在 Phase 1 引入复杂权限系统或多租户云 RBAC
- 不在 Phase 1/2 强制上传任何本地统计数据到远端
- 不在 Phase 1 构建实时流式监控平台
- 不在 Phase 1/2 依赖外部 BI 工具才能使用核心功能

## 3. 核心功能需求

### 3.1 多项目 / 多 agent / 多成员视角

Dashboard 必须支持团队维度下的多种切片与聚合。

基础聚合维度：

- `project`
- `agent`
- `session`
- `member`（若本地记录可推导为用户名、机器名或显式配置的团队成员 ID）
- `tool` / `op_type`
- 时间窗口（天 / 周 / 月 / 自定义区间）

典型视图：

- 团队总览：全团队总节省、总 API 成本、净节省率
- 项目视图：每个项目的节省排名、成本占比、趋势变化
- 成员视图：每位成员的调用量、节省量、节省率
- agent 视图：不同 agent 类型的节省表现
- 会话视图：定位高成本或高节省的具体 session

#### 需求细化

- 支持 Top N 排名与全文量汇总
- 支持按时间窗口筛选后再按维度 drill-down
- 支持多维组合过滤，例如“某项目 + 某成员 + 近 30 天”

### 3.2 按成员 / 会话 / 工具 / 项目查看节省

系统必须至少展示以下指标：

- `before_tokens`
- `after_tokens`
- `tokens_saved`
- `savings_rate`（`tokens_saved / before_tokens`）
- 估算 API 成本
- 节省折算金额

分维度要求：

#### 成员维度

- 每位成员累计节省 token
- 每位成员累计节省金额
- 每位成员平均会话节省率
- 每位成员高峰日/高峰周识别

#### 会话维度

- 单个 session 的请求数
- 单个 session 的压缩前后 token 总量
- 单个 session 的节省金额
- session 内工具 / agent 构成

#### 工具维度

工具可指 hook/op type/tool family，例如：

- `compress-schema`
- `compress-response`
- `rewrite-command`
- 后续新增 MCP / dashboard export / semantic compression 等类别

展示要求：

- 各工具累计节省量与占比
- 各工具平均单次节省效果
- 各工具趋势变化

#### 项目维度

- 项目总请求量
- 项目总 token 消耗
- 项目总节省与净节省率
- 项目内成员贡献拆分

### 3.3 token 与 API 成本联动

Dashboard 必须将 token 指标与货币指标统一展示。

基准定价：

- 默认采用 `$3 / 1M tokens` 的统一换算口径用于报表与预算视图
- 该口径作为**默认估算值**，后续可替换为模型级/供应商级实际价格表

基础公式：

```text
estimated_cost_usd = total_tokens / 1_000_000 * 3
estimated_saved_usd = saved_tokens / 1_000_000 * 3
```

同时需要兼容未来扩展：

- 不同模型不同单价
- 输入 / 输出 token 分别定价
- 读取已有 `pricing_snapshot_json` 或后续价格配置

展示层至少提供：

- 总 token 节省
- 总 API 成本估算
- 节省折算金额
- 节省率与金额率

### 3.4 周 / 月趋势图

Dashboard 必须提供时间趋势能力，用于回答“节省是否可持续”。

至少支持：

- 最近 7 天
- 最近 30 天
- 按周汇总
- 按月汇总

建议图表：

- token 消耗趋势折线图
- token 节省趋势折线图
- 节省金额趋势柱状图
- 项目或成员分组堆叠图

趋势图需要支持：

- 同时展示“压缩前”“压缩后”“节省量”三条指标
- 支持导出 PNG 或嵌入静态 HTML 报告
- 对缺失数据区间进行零值或空值策略的显式处理，避免误读

### 3.5 异常峰值检测

Dashboard 必须具备基础异常识别能力，用于发现异常成本和异常节省波动。

首期可采用简单、可解释的启发式规则：

- 某时间桶 token 消耗超过过去 N 个同类时间桶平均值的 2x 或 3x
- 单个 session 的 token 使用量超过团队 P95 / P99
- 某成员、项目、工具在短窗口内出现突增
- 节省率异常下降，例如从常态 40% 降到 5%

异常输出应包含：

- 异常对象（项目 / 成员 / session / 工具）
- 触发时间
- 异常类型（高成本 / 高峰值 / 低节省率）
- 触发规则说明
- 建议下一步动作（如检查 prompt、压缩配置、工具行为）

### 3.6 预算预警

Dashboard 必须允许配置预算阈值，并在逼近或超出预算时给出提示。

预算粒度建议至少包括：

- 团队月预算
- 项目月预算
- 成员月预算（可选）

预警阈值建议：

- 50%：早期关注
- 80%：预警
- 100%：超预算

展示要求：

- 当前预算使用率
- 按当前趋势预测的月末支出
- 哪些项目或成员是主要贡献者
- 对应的节省金额与“若无 tokenless 可能支出多少”的对比

## 4. 数据架构

### 4.1 当前数据基础

当前 `stats.db` 已具备团队面板的关键原始维度基础，至少包括或可推导出：

- `project`
- `agent`
- `session`
- 时间戳
- 压缩前后 token / bytes
- 节省 token / bytes
- 操作类型 / 方法

另外，结合 `0018` 中定义的统计联动能力，系统可逐步补充：

- API 成本记录
- pricing snapshot
- session 与上游请求关联

因此团队面板的核心不是重新采集，而是：

1. 统一查询口径
2. 建立聚合层
3. 提供导出与展示层

### 4.2 聚合查询层

需要在本地统计之上建立聚合查询能力，支持：

- 按天 / 周 / 月分桶
- 按项目 / 成员 / agent / 工具 / session 聚合
- 统计总量、均值、P95、异常阈值
- 产出适合前端消费的 JSON 结构

建议引入两层数据访问模式：

#### 明细层

面向排障与 drill-down：

- 单条操作记录
- 单 session 明细
- 单成员明细

#### 汇总层

面向 dashboard 视图：

- 团队总览 KPI
- 趋势序列
- 排行榜
- 预算与异常视图

### 4.3 导出能力

需要支持本地导出，至少包括：

- JSON 导出：供静态 HTML 或其他工具消费
- CSV 导出：供财务、管理者、电子表格分析使用
- HTML 报告导出：供直接浏览、归档、分享

建议导出内容：

- `summary.json`
- `trends.json`
- `projects.json`
- `members.json`
- `alerts.json`
- `report.html`

### 4.4 可选的中央汇聚服务

为 Phase 3 做准备，可设计可选中央汇聚服务，但不得影响 Phase 1/2 的本地优先策略。

中央汇聚服务的目标：

- 汇聚多台开发机、多团队、多项目的数据
- 实现跨组织报表
- 提供托管 dashboard 与共享链接
- 支持更细粒度权限与 SaaS 商业化

约束：

- Phase 1/2 不依赖该服务
- 所有上传应为显式启用
- 上传粒度、脱敏策略、保留期必须可配置

## 5. 分阶段实现

### 5.1 Phase 1 — 本地 JSON 导出 + 静态 HTML 报告

目标：在不引入长驻服务的前提下，最快交付团队级展示能力。

建议入口：

- `tokenless stats export --team-json`
- `tokenless stats export --team-html`
- 或新增 `tokenless report team`

Phase 1 交付内容：

- 从 `stats.db` 读取并聚合
- 导出标准 JSON 文件集
- 生成静态 HTML 报告
- 报告内置周/月趋势图、排行榜、预算视图、异常列表
- 可离线打开，无需网络

适用场景：

- 管理汇报
- 销售演示
- 试点团队 ROI 验证
- 用户把本地报告发给内部干系人

优点：

- 实现成本低
- 不引入后台服务
- 安全边界清晰，数据仍在本地

限制：

- 不是实时交互式应用
- 跨机器团队数据需要手动汇总
- 预算告警以导出时计算为主

### 5.2 Phase 2 — 本地轻量 Web 服务器（`tokenless serve`）

目标：提供本地持续运行的交互式 dashboard。

建议入口：

- `tokenless serve`
- 默认绑定 `127.0.0.1`
- 默认仅本机访问

Phase 2 交付内容：

- 本地 HTTP 服务
- Dashboard Web UI
- 交互式筛选、排序、钻取
- 实时或准实时刷新统计数据
- 预算告警页面与异常详情页面
- 本地导出 JSON / CSV / HTML

优点：

- 用户体验显著提升
- 更适合日常查看与排障
- 为 Phase 3 的托管版前后端分层打基础

新增考虑：

- 本地服务生命周期管理
- 浏览器访问安全
- 本地缓存与分页查询

### 5.3 Phase 3 — SaaS 托管版

目标：将团队 dashboard 升级为可共享、可集中治理、可订阅的商业产品。

Phase 3 交付方向：

- 可选云同步 / 数据上报
- 多团队、多租户、多项目集中查看
- 成员与组织权限模型
- 共享报表、统一预算策略、组织级告警
- 集中式异常检测与长周期趋势分析

商业化价值：

- 订阅制团队版入口
- 向 CTO / 平台团队提供统一成本控制能力
- 形成从 CLI 工具到团队平台的升级路径

主要挑战：

- 数据采集与隐私合规
- 多租户隔离
- 身份认证与访问控制
- 更复杂的价格模型和账单一致性

## 6. 安全与隐私

### 6.1 Phase 1 / 2 原则：数据不出本地

Phase 1 与 Phase 2 默认必须满足：

- 所有统计数据仅保存在本地
- 默认不进行任何网络上传
- HTML 报告、JSON、CSV 导出全部写入本地文件系统
- 本地 Web 服务默认仅绑定回环地址

### 6.2 Phase 3 原则：云同步必须显式启用

若进入 SaaS 托管版，必须遵循：

- 云同步默认关闭
- 用户显式配置后才允许上传
- 上传范围、脱敏字段、保留时间可配置
- 应支持仅上传汇总、不上传原始明细的模式

### 6.3 敏感数据最小化

团队面板中原则上不应依赖或暴露 prompt 正文、响应正文等高敏内容。

推荐只使用以下低敏统计字段：

- token 数量
- 时间戳
- 项目标识
- agent 类型
- 操作类型
- 成员标识（可匿名化或哈希化）
- 成本估算字段

### 6.4 访问与导出安全

对于本地服务与导出能力，应考虑：

- 导出路径安全校验
- 本地服务禁止默认公网监听
- 报告中避免包含原始 secrets、prompt、响应正文
- 若支持共享 HTML，应明确提示其可能包含组织成本信息

## 7. 竞品参考

本功能定位介于“开发者工具统计面板”与“FinOps 成本看板”之间，可参考以下竞品与相邻产品方向：

### 7.1 LLM / AI 成本管理类产品

- Helicone
- Langfuse
- Humanloop（偏评估与观测）
- OpenMeter / usage metering 类产品

可借鉴点：

- usage 趋势可视化
- 按项目 / key / user 聚合
- 成本归因与预算告警

### 7.2 工程团队可观测 / FinOps 工具

- Datadog cost / usage dashboards
- Grafana dashboards
- Cloud provider budget dashboards
- Vercel / OpenAI usage dashboards

可借鉴点：

- 周期趋势图
- 异常峰值标注
- 预算阈值与预测支出

### 7.3 tokenless 的差异化优势

与通用 LLM 观测平台相比，tokenless 的差异化不只是“花了多少”，而是：

- “原本会花多少”
- “实际只花了多少”
- “中间具体是哪些压缩/改写策略在节省”

也就是说，tokenless 不只是成本观测，而是**成本压缩收益证明工具**。Team Savings Dashboard 应突出这种 ROI 叙事：

- 成本可见
- 节省可证
- 策略可归因
- 团队价值可汇报

## 8. 路线图

### 8.1 近期里程碑

#### Milestone A — 数据模型对齐

- 明确 `stats.db` 中团队报表所需字段
- 补齐成员标识来源与 session 归因策略
- 与 `0018` 的成本记录字段统一命名与换算口径

#### Milestone B — 聚合与导出 CLI

- 增加 team summary 聚合查询
- 支持 JSON / CSV 导出
- 输出稳定 schema，供 HTML 报告消费

#### Milestone C — 静态 HTML 报告

- 提供总览、趋势、排行榜、异常、预算模块
- 支持离线查看
- 更新文档与示例截图

### 8.2 中期里程碑

#### Milestone D — `tokenless serve`

- 本地 Web 服务
- REST/JSON 接口或内嵌 API
- 交互式筛选与 drill-down
- 预算与异常实时刷新

#### Milestone E — 高级分析能力

- 更细粒度峰值检测
- 预算预测
- 同期对比（WoW / MoM）
- 导出更多财务报表格式

### 8.3 长期里程碑

#### Milestone F — SaaS 团队版

- 中央汇聚服务
- 多租户组织管理

#### Milestone G — 商业化包装

- Free / Team / Enterprise 分层
- 团队预算控制与治理能力作为付费卖点
- 结合压缩收益证明输出销售演示材料

## 9. 成功指标

Team Savings Dashboard 的效果可用以下指标评估：

- 团队用户能否在 5 分钟内看懂“节省了多少钱”
- 是否能按项目 / 成员 / 会话快速定位异常高成本来源
- 是否能导出可用于汇报的静态报告
- 是否能支撑试点团队的 ROI 证明
- 是否提高 tokenless 团队版的付费转化率

建议产品 KPI：

- 团队报表导出使用率
- 周/月趋势查看频率
- 预算预警触发后的用户回访率
- 从单用户 CLI 升级到团队 dashboard 的转化率

---

## 10. 总结

Team Savings Dashboard 是 tokenless 从“个人效率工具”走向“团队级成本控制产品”的关键一步。

它复用了现有统计与成本联动基础，不要求一开始就上云，而是采用清晰的三阶段策略：

1. 本地导出 + 静态 HTML 报告
2. 本地交互式 Web 控制台
3. 可选 SaaS 托管版

这一设计既符合当前本地优先、安全优先的产品原则，也为后续商业化留出足够演进空间。

如果 `0018` 解决了“如何把压缩与成本关联起来”，那么 `0020` 解决的就是“如何把这种节省价值解释给团队、管理者与付费决策者看”。
